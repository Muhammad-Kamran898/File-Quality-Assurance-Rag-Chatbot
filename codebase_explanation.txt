1. PROJECT OVERVIEW (IN SIMPLE WORDS)
================================================================================

This is a PDF Question-Answering App. Here's what it does in simple terms:

    USER UPLOADS PDFs  -->  APP READS PDFs  -->  APP CREATES "SMART SEARCH"
         |                    |                      |
    User drops        App extracts text     App converts text to
    PDF files         from PDFs             numbers (embeddings)
                                                    |
                                                    v
    USER ASKS QUESTIONS  <--  APP FINDS ANSWERS  <--  APP SEARCHES
         |                    |                      |
    "What is this         App shows answer      App finds relevant
    document about?"      with sources           text chunks

Think of it like:
- You have a super-smart assistant who has read all your PDFs
- When you ask questions, it finds the exact pages that answer your question
- It then uses AI to summarize the answer in a human-like way

SECTION 3: CONFIGURE_RETRIEVER FUNCTION (Lines 37-100)
------------------------------------------------------------------------------

@st.cache_resource(ttl="1h", show_spinner=False)
def configure_retriever(uploaded_files):
    # Create the embedding model (converts text to numbers)
    embeddings_model = OllamaEmbeddings(
        model="nomic-embed-text",
        base_url="http://localhost:11434"
    )

    # Where to save the database
    persist_dir = "./chroma_db_store"


    # ---------- CASE 1: USER UPLOADED NEW FILES ----------
    if uploaded_files:
        docs = []
        temp_dir = tempfile.TemporaryDirectory()

        # Step 1: Save each uploaded file temporarily
        for file in uploaded_files:
            temp_filepath = os.path.join(temp_dir.name, file.name)
            with open(temp_filepath, "wb") as f:
                f.write(file.getvalue())

            # Step 2: Read/load the PDF
            loader = PyMuPDFLoader(temp_filepath)
            docs.extend(loader.load())

        # Step 3: Split text into chunks (1500 chars each, 200 overlap)
        text_splitter = RecursiveCharacterTextSplitter(
            chunk_size=1500,
            chunk_overlap=200
        )
        doc_chunks = text_splitter.split_documents(docs)

        # Error check: did we extract any text?
        if not doc_chunks:
            st.error("No text could be extracted from the uploaded PDF.")
            st.stop()

        # Step 4: Create the vector database
        st.info("Building new database and saving to disk...")

        vector_db = Chroma(
            persist_directory=persist_dir,
            embedding_function=embeddings_model
        )

        # Step 5: Add chunks to database in batches (with progress bar)
        total_chunks = len(doc_chunks)
        batch_size = 100
        progress_text = f"Embedding {total_chunks} chunks..."
        my_bar = st.progress(0, text=progress_text)

        for i in range(0, total_chunks, batch_size):
            batch = doc_chunks[i : i + batch_size]
            vector_db.add_documents(batch)
            current_progress = min((i + batch_size) / total_chunks, 1.0)
            chunks_processed = min(i + batch_size, total_chunks)
            my_bar.progress(
                current_progress,
                text=f"Processed {chunks_processed} out of {total_chunks} chunks...",
            )

        my_bar.empty()
        return vector_db.as_retriever()


    # ---------- CASE 2: NO NEW FILES - LOAD EXISTING DATABASE ----------
    else:
        if os.path.exists(persist_dir) and os.listdir(persist_dir):
            st.success("Loaded existing database from disk!")
            vector_db = Chroma(
                persist_directory=persist_dir,
                embedding_function=embeddings_model
            )
            return vector_db.as_retriever()
        else:
            st.info("No existing database found. Please upload a PDF in the sidebar.")
            st.stop()

WHAT THIS DOES IN SIMPLE TERMS:

1. @st.cache_resource(ttl="1h") - Cache this function for 1 hour so it doesn't
   reload every time the user interacts with the app

2. OllamaEmbeddings - Converts text into "vectors" (lists of numbers)
   Each piece of text becomes a point in mathematical space

3. Chunk Size 1500 - We cut the PDF into pieces of 1500 characters
   Overlap 200 - Each chunk overlaps with the previous one by 200 chars
   This ensures context isn't lost at chunk boundaries

4. Batch Processing - Instead of adding all chunks at once (which could crash),
   we add 100 at a time with a progress bar

5. ChromaDB - Stores all these vectors on disk in "chroma_db_store" folder

SECTION 4: STREAM HANDLER - FOR REAL-TIME TEXT DISPLAY (Lines 103-114)
------------------------------------------------------------------------------

    class StreamHandler(BaseCallbackHandler):
        def __init__(self, container, initial_text=""):
            self.container = container
            self.text = initial_text
            self.ctx = get_script_run_ctx()

        def on_llm_new_token(self, token: str, **kwargs) -> None:
            add_script_run_ctx(ctx=self.ctx)
            self.text += token
            self.container.markdown(self.text)

WHAT THIS DOES:

- When the AI generates text, it comes out token by token (word by word)
- This class captures each token and displays it immediately on the screen
- Without this, you'd have to wait for the entire answer before seeing anything
- get_script_run_ctx() and add_script_run_ctx() are needed because the AI
  runs in a background thread, and we need to connect it to the UI thread

SECTION 5: POST MESSAGE HANDLER - FOR SOURCE ATTRIBUTION (Lines 117-146)

    class PostMessageHandler(BaseCallbackHandler):
        def __init__(self, container):
            BaseCallbackHandler.__init__(self)
            self.container = container
            self.sources = []

        def on_retriever_end(self, documents, *, run_id, parent_run_id, **kwargs):
            source_ids = []
            for d in documents:
                source = d.metadata.get("source", "Unknown")
                page = d.metadata.get("page", "Unknown")

                metadata = {
                    "source": source,
                    "page": page,
                    "content": d.page_content[:200] + "...",
                }
                idx = (source, page)
                if idx not in source_ids:
                    source_ids.append(idx)
                    self.sources.append(metadata)

        def on_llm_end(self, response, *, run_id, parent_run_id, **kwargs):
            if len(self.sources):
                with self.container:
                    st.markdown("**Sources:**")
                    st.dataframe(
                        data=pd.DataFrame(self.sources[:3]),
                        use_container_width=True
                    )

WHAT THIS DOES:

- After the retriever finds relevant document chunks, this captures WHICH
  documents and WHICH pages were found
- It stores the source filename and page number
- After the AI finishes generating the answer, it displays a table showing
  the top 3 sources used to answer the question

SECTION 11: HANDLE USER INPUT (Lines 216-238)
------------------------------------------------------------------------------

    if user_prompt := st.chat_input():
        st.chat_message("human").write(user_prompt)

        with st.chat_message("ai"):
            # Create placeholder for streaming text
            response_placeholder = st.empty()

            # Create container for sources
            sources_container = st.container()

            # Initialize both callbacks
            stream_handler = StreamHandler(response_placeholder)
            pm_handler = PostMessageHandler(sources_container)

            # Configure callbacks
            config = {"callbacks": [pm_handler, stream_handler]}

            # Run the chain!
            response = qa_rag_chain.invoke(
                {"input": user_prompt, "chat_history": streamlit_message_history.messages},
                config,
            )

WHAT THIS DOES:

- Waits for user to type a question
- Displays user's question in the chat
- Creates empty placeholders for AI response and sources
- Sets up both handlers (one for streaming, one for sources)
- Runs the entire RAG chain and displays results

5. HOW EVERYTHING WORKS TOGETHER
================================================================================

STEP-BY-STEP WALKTHROUGH:

1. APP STARTS
   - Streamlit creates the web page
   - If no PDF uploaded, loads existing database from disk
   - Shows welcome message

2. USER UPLOADS PDF
   - Files appear in sidebar
   - App detects new files
   - Reads PDF text, splits into chunks, creates embeddings
   - Saves everything to ChromaDB
   - Shows progress bar during processing

3. USER ASKS QUESTION
   - Types in chat input box
   - Question appears in chat

4. APP PROCESSES QUESTION
   a) Reformulates question using chat history (if needed)
   b) Converts question to embedding vector
   c) Searches ChromaDB for similar vectors
   d) Gets top matching text chunks
   e) Sends chunks + question to Ollama LLM
   f) LLM generates answer (token by token)
   g) StreamHandler shows answer as it's generated
   h) PostMessageHandler shows source documents

5. USER ASKS ANOTHER QUESTION
   - Chat history is updated
   - If question references previous context, it's reformulated
   - Process repeats

9. KEY CONCEPTS EXPLAINED
================================================================================

WHAT IS RAG?
    RAG = Retrieval-Augmented Generation
    Instead of the AI making up answers, it:
    1. Finds relevant document parts (retrieval)
    2. Uses them as context (augmentation)
    3. Generates answer (generation)

WHAT IS AN EMBEDDING?
    An embedding is a list of numbers that represents the "meaning" of text.
    Similar concepts have similar numbers.
    Example: "cat" and "dog" have similar embeddings (both are pets)
    "cat" and "car" have different embeddings

WHAT IS A VECTOR DATABASE?
    A database that stores embeddings instead of regular data.
    Allows "semantic search" - finding similar meanings, not just exact matches.

WHAT IS A RETRIEVER?
    Something that finds relevant documents from a database.
    Takes a question, returns the most relevant text chunks.

WHAT IS A CHAIN?
    A series of connected AI components.
    Data flows through each component step by step.

WHAT IS CALLBACK HANDLER?
    Code that runs at specific points during AI processing.
    - StreamHandler: Runs every time a new token is generated
    - PostMessageHandler: Runs after retrieval and after generation
